# Red-team review ‚Äî 2026-04-26 (post-PR-219)

> Historical assessment: the hash-echo approval mechanism described here was
> replaced in July 2026 by transaction-bound, one-time approval receipts. See
> [SECURITY.md](../../SECURITY.md) for the current protocol and trust boundary.

## Threat model

The attacker has obtained (1) a shell account in the Linux groups that grant a
`CallerRole` of Dev or Admin under `policy.rs` ‚Äî either via `sysknife-dev`,
`sysknife-admin`, or `wheel` ‚Äî and (2) the ability to write arbitrary framed
JSON to the daemon's Unix socket or vsock. The daemon runs as the `sysknife`
system user and sits above `sudo` via `/etc/sudoers.d/sysknife`. The attacker's
goal is to execute arbitrary commands as root, to suppress or forge audit records,
or to exfiltrate credentials.

The daemon's two core defences are: the typed-action surface (every IPC request
names an action and the daemon builds the argv internally, never from raw strings),
and the then-current request/approval hash contract that prevented a signed
preview from authorising a different payload. All attacker-controlled strings pass through
dedicated validators (`validated_safe_arg`, `validated_username`, `IpAddr::from_str`,
etc.) before reaching argv. The review examined whether the new Ubuntu surface,
the sudoers additions, the `GrubSetKargs` bash pipeline, and the `ProAttach` redaction
layer hold under crafted inputs.

## Findings (CRITICAL)

No CRITICAL findings. Every plausible privilege-escalation path examined was
stopped by one of the layered defences described below.

## Findings (HIGH)

### HI1. `sudo env` is NOPASSWD ‚Äî attacker can enumerate env-var positions to inject an arbitrary binary via `NetplanSet key=PATH value=‚Ä¶` ‚úÖ Fixed

> **Resolved (July 2026):** the `env` grant is now constrained to
> `/usr/bin/env DEBIAN_FRONTEND=noninteractive NEEDRESTART_MODE=a /usr/bin/apt-get *`
> in `packaging/sysknife-sudoers` ‚Äî the unrestricted `env` NOPASSWD foothold is gone.

- **File:** `packaging/sysknife-sudoers:69`, `crates/sysknife-daemon/src/actions/netplan.rs`
- **Adversary input:** `{"type":"preview","action_name":"NetplanSet","params":{"key":"<something>","value":"<something>"}}`; then the matching execute.
- **Daemon behavior:** `netplan set <key> <value>` is called; it does NOT use `sudo env`. `NetplanApply` also does not use `sudo env`. The `env` NOPASSWD entry is used exclusively by `apt.rs` to set `DEBIAN_FRONTEND` / `NEEDRESTART_MODE` ahead of the `apt-get` binary. The `key` and `value` params are both `validated_safe_arg`-gated ‚Äî they cannot contain spaces, `=`, `$`, or any shell metacharacter ‚Äî so an attacker cannot inject additional `KEY=VAL` arguments or change the binary that `env` ultimately invokes.
- **Impact (mitigated):** The theoretical scenario is: craft `key=DEBIAN_FRONTEND` and `value=noninteractive /usr/bin/id` to append an extra word that `env` would interpret as a program. This does NOT work here because `validated_safe_arg` rejects the space in `noninteractive /usr/bin/id`, and because `NetplanSet` calls `netplan` directly (no `env` wrapper).
- **Residual risk (HIGH):** The `sudo env` NOPASSWD line is **unconstrainted by path** ‚Äî it allows `sysknife` to run `sudo env <anything>`. If any future action is added that constructs the `env`-prefixed argv with insufficiently validated arguments, an attacker could leverage the open `env` grant to pivot to an arbitrary binary. The current code has no such path, but the grant is a standing foothold.
- **Fix:** Restrict the sudoers `env` line to only the specific invocations that actually occur:
  ```
  sysknife ALL=(root) NOPASSWD: /usr/bin/env DEBIAN_FRONTEND=noninteractive NEEDRESTART_MODE=a apt-get *
  ```
  Wildcards in sudoers are coarse but this is still substantially narrower than an unrestricted `env` NOPASSWD grant.

### HI2. `sudo python3` is NOPASSWD with no argument constraint ‚Äî arbitrary root code execution if `GrubSetKargs` argv can be coerced ‚úÖ Fixed

> **Resolved (July 2026):** the open `/usr/bin/python3` grant was removed. The
> GrubSetKargs logic now lives in a root-owned helper, granted NOPASSWD only for
> `/usr/lib/sysknife/grub-kargs-edit *` (`packaging/sysknife-sudoers`) ‚Äî the
> exact architectural fix recommended below.

- **File:** `packaging/sysknife-sudoers:117`, `crates/sysknife-daemon/src/actions/grub.rs:147-163`
- **Adversary input:** `GrubSetKargs` with `append=["quiet"]` and `delete=["splash"]` (both pass `validated_safe_arg`).
- **Daemon behavior:** The action builds a `bash -c <script>` string where `<script>` embeds `sudo python3 -c '<python_script>'`. The python script is constructed by string interpolation of `append_str` and `delete_block`. Both `append` and `delete` items are already `validated_safe_arg`-gated (no quotes, no `$`, no backslash, no semicolons), so the interpolated Python script body is safe. The `{:?}` Rust debug-formatter used for `script` in `shell_cmd` adds double-quotes around the entire script but does NOT escape embedded double quotes in the script's own Python string literals ‚Äî however Python string literals there use single quotes (`'‚Ä¶'`). Validated args cannot contain `'`, so the outer `{:?}` wrapping is safe.
- **Residual risk (HIGH):** The sudoers line `sysknife ALL=(root) NOPASSWD: /usr/bin/python3` with **no argument restriction** means that if any other code path (a future action, a bug in the dispatch, a shell escape not currently present) passes attacker-controlled bytes to `python3 -c '‚Ä¶'`, full root code execution follows. The validator closure is the only guard.
- **Fix:** Add argument-level constraints in sudoers:
  ```
  sysknife ALL=(root) NOPASSWD: /usr/bin/python3 -c *
  ```
  Or (better) move the GrubSetKargs Python logic into a small dedicated script owned by root (`/usr/lib/sysknife/grub-kargs-edit`) and grant NOPASSWD only to that script. This eliminates the open python3 grant entirely.

### HI3. `sudo cp` is NOPASSWD with no path restriction ‚Äî arbitrary root file copy if cp argv can be coerced ‚úÖ Fixed

> **Resolved (July 2026):** the unconditional `/usr/bin/cp` grant was removed;
> GrubSetKargs no longer shells out to `sudo cp` (the backup/restore of
> `/etc/default/grub` moved into the `grub-kargs-edit` helper).

- **File:** `packaging/sysknife-sudoers:116`, `crates/sysknife-daemon/src/actions/grub.rs:147-151`
- **Adversary input:** Same as HI2 ‚Äî the shell fragment invokes `sudo cp <source> <dest>`. Both source and destination paths are hardcoded constants (`GRUB_DEFAULT`, `GRUB_BACKUP`). No attacker-controlled string reaches these positions.
- **Residual risk (HIGH):** The sudoers grant `sysknife ALL=(root) NOPASSWD: /usr/bin/cp` is unconditional. A future action that builds a shell string containing `sudo cp <attacker_path> <dest>` would give the attacker a root-owned copy of any file.
- **Fix:**
  ```
  sysknife ALL=(root) NOPASSWD: /usr/bin/cp /etc/default/grub /etc/default/grub.sysknife.bak, \
                                /usr/bin/cp /etc/default/grub.sysknife.bak /etc/default/grub, \
                                /usr/bin/cp /etc/default/grub.sysknife.bak /etc/default/grub.tmp
  ```
  Enumerate only the exact copies that `GrubSetKargs` makes.

## Findings (MEDIUM)

### ME1. `GrubSetKargs` bash pipeline runs under plain `bash -c` via `sudo sh` ‚Äî not under the restricted-path sudoers entry

- **File:** `crates/sysknife-daemon/src/actions/grub.rs:158-163`
- **Detail:** `GrubSetKargs` uses `command_mechanism("bash", ["-c", &shell_cmd])`. The daemon invokes `bash` directly (not via `sudo bash`), then the individual `sudo cp`, `sudo python3`, `sudo update-grub` calls inside the fragment each hit their own sudoers lines. However `/usr/bin/sh` has a NOPASSWD sudoers grant. If a future action mistakenly routes through `sudo sh -c <attacker_string>` instead of through the typed action surface, this grant combined with ME1 is dangerous.
- **Impact:** Defense-in-depth gap only; no current code path triggers it.
- **Fix:** Remove `/usr/bin/sh` from sudoers if it is no longer needed by any Fedora action, or document exactly which actions use it.

### ME2. `redact_argv` does VALUE-MATCH only ‚Äî token equal to structural arg `"attach"` suppresses that structural element ‚úÖ Fixed in #216

- **File:** `crates/sysknife-daemon/src/dispatcher.rs:84-105`
- **Detail:** If an operator issues `ProAttach` with `token = "attach"`, `redact_argv` replaces BOTH the structural `"attach"` and the token position with `"<REDACTED>"`. The DescribeResponse command string becomes `sudo pro <REDACTED> <REDACTED>` instead of `sudo pro <REDACTED>`. This is confusing but not exploitable ‚Äî the redaction is cosmetic only (the rendered string is never used for execution), and `validated_safe_arg` does not block the string `"attach"` (alphanumeric). A caller using this intentionally gets a misleading display but no escalation.
- **Fix (applied):** `credential_argv_position("ProAttach")` returns `Some(usize::MAX)` (last element). `redact_argv` applies positional redaction first ‚Äî the token is always replaced by index regardless of its value ‚Äî then falls back to value-match only for actions without a positional spec. This closes the structural-arg suppression edge case. See `dispatcher.rs:81-115` ("Strategy (ME2 fix)" comment).

### ME3. Audit log tail truncation is undetectable by chain walk ‚úÖ Fixed (closes #223)

- **File:** `crates/sysknife-daemon/src/audit_chain.rs:24-25` (documented in module comment)
- **Detail:** An attacker with write access to the SQLite database file can DELETE the last K rows and the chain remains valid (no forward reference from the deleted tail). The module comment acknowledges this: "Tail truncation is undetectable by chain walk alone."
- **Fix (applied):** `crates/sysknife-daemon/src/audit_watermark.rs` ‚Äî every successful SQLite INSERT emits a structured journald line under syslog identifier `sysknife-audit-tip` with body `seq=<N> chain_hash=<hex64>`. A SIEM compares the journal stream against the SQLite tail: any journal entry beyond the SQLite tail = truncation occurred. Failure to write to journald is non-fatal (one-time stderr warning via `OnceLock`). 8 new tests, all passing.

### ME4. `UbuntuReleaseUpgrade` 20‚Äì45 min execution window ‚Äî no concurrency guard at the action level ‚úÖ Fixed (closes #224)

- **File:** `crates/sysknife-daemon/src/dispatcher.rs:1285-1305` (claim_for_execution)
- **Detail:** `claim_for_execution` atomically transitions a single transaction from `Queued ‚Üí Running`, preventing two concurrent executes of the SAME request hash. However, a second IPC client can issue a DIFFERENT action (e.g., `AptInstall`) while `do-release-upgrade` is still running, potentially creating dpkg lock contention or corrupting an in-flight upgrade. There is no global "upgrade in progress" mutex.
- **Impact:** Denial-of-service / upgrade corruption. Not a privilege escalation.
- **Fix (applied):** `DaemonState.running_high_risk_reboot: Arc<Mutex<Option<String>>>` holds the `request_hash` of any in-flight High-risk + reboot-required action. Any new mutating action checks the slot before `claim_for_execution`; if occupied, returns `DaemonResponse::ConflictResponse`. Read-only (Observer-level) actions bypass the check entirely. Slot is cleared after execute completes (success or failure). 5 new integration tests. Known open: no timeout ‚Äî a hung action holds the slot indefinitely; daemon restart required to recover.

### ME5. AppArmor profile path passes `validated_safe_arg` ‚Äî `/` is allowed, enabling traversal to any profile path ‚úÖ Fixed in #220

- **File:** `crates/sysknife-daemon/src/executor.rs:711-719`
- **Detail:** `validated_safe_arg` explicitly permits `/`. An `AppArmorEnforce` request with `profile_path = "/etc/apparmor.d/../../../../etc/cron.d/evil"` would be blocked by `..` not being in the allowlist (`.` is allowed but sequential `..` contains no non-allowlist chars). Wait ‚Äî `..` is `[`.`][`.`]`: both dots individually pass `validated_safe_arg`. The leading-dash check does not apply. The `/` allows path separators. So `../etc/passwd` survives `validated_safe_arg`. However the actual binary called (`aa-enforce <path>`) processes only AppArmor policy files; it will simply fail on a non-policy file without becoming a privilege escalation vector. The concern is informational leakage: if `aa-enforce /etc/shadow` outputs the file content in an error, the attacker learns shadow file contents.
- **Fix (applied):** `validated_apparmor_profile()` in `validate.rs` requires the path to start with `/etc/apparmor.d/` or `/usr/share/apparmor/` and passes the remainder through `validated_safe_arg`. Path traversal via `..` is blocked because `.` is in the allowlist but `..` as a path component is rejected by the prefix requirement (the prefix itself must be an exact match at the start).

## Confirmed clean

### Request hash ‚Üí approval hash binding is tight

`compute_request_hash` in `dispatcher.rs:396-408` hashes `action_name || NUL || canonical_json(params)`. `canonical_json` sorts object keys recursively. An attacker who tries to swap params after preview (e.g., change `package` from `curl` to `malware`) produces a different hash and is rejected. The `find_by_request_hash` + `claim_for_execution` pair closes the TOCTOU window: a second concurrent execute of the same hash after the first claims it returns `claimed = false` and a `stale_approval` error. **This is tight.**

### ProAttach token redaction is complete at the wire boundary

The token is redacted in `PreviewEnvelope.proposed_change` (via `redact_params`), in `DescribeResponse.command` (via `redact_argv`), and in `RequestEnvelope.params` never crosses the wire at all (the `RequestEnvelope` is only constructed in-process). The audit log's `summary` field uses `PreviewProfile`-derived text that does not include params. The SIEM forwarder (`forward_audit_event`) reads chain-row fields that also contain only the redacted summary. The token survives only in the executor's argv at the point `execute_spec` calls `tokio::process::Command::new("sudo").args(‚Ä¶)` ‚Äî that is the intended single clear-text exposure. **No token leakage path found.**

### `validated_safe_arg` unicode bypass ‚Äî no path found

The allowlist in `validate.rs:148` is `is_ascii_alphanumeric || matches!(c, '.' | '_' | ':' | '/' | '+' | '@' | '-')`. The Rust `char::is_ascii_alphanumeric` returns false for any non-ASCII codepoint. Unicode lookalikes (U+0430 Cyrillic `–∞` vs ASCII `a`) fail `is_ascii_alphanumeric`. RTL override marks (U+202E) are non-ASCII. There is no downstream `to_lowercase` or Unicode normalization step that could reclassify a homoglyph to an ASCII metacharacter. **Tight.**

### vsock auth token ‚Äî replay and embedded-null attacks

`auth.rs:64` rejects the empty string. The length-mismatch shortcut in the constant-time comparator fires before `ct_eq` on different-length inputs ‚Äî correct. A token with embedded null bytes in the JSON string would be decoded by `serde_json` without the null, producing a different effective token that won't match the stored value. The 10-second `VSOCK_AUTH_FRAME_TIMEOUT_SECS` closes the slow-loris slot against partially-written frames. `MAX_MESSAGE_BYTES = 4 MiB` prevents memory exhaustion from a crafted large-length header. **Tight.**

### `GrubSetKargs` Python regex-escape completeness

`regex_escape` in `grub.rs:172-181` escapes `. ^ $ * + ? ( ) [ ] { } | \`. The escaped string is placed inside a Python `re.sub(r'...')` raw string literal delimited by `'...'`. The validated args cannot contain `'` (single-quote is not in the `validated_safe_arg` allowlist and fails the metacharacter check at `validate.rs:427`). Therefore an attacker cannot close the raw-string literal and inject Python code. The missing escapes for `'` and `"` are not exploitable given the validator. **Tight within the current validator.**

### `Fail2banBanIp` / `Fail2banUnbanIp` IP validation

The `ip` param is passed to `IpAddr::from_str` at `fail2ban.rs` before reaching `fail2ban-client set <jail> banip <ip>`. `IpAddr::from_str` rejects every string that is not a well-formed IPv4 or IPv6 address ‚Äî no shell metacharacter survives. Jail name is `validated_safe_arg`-gated. **Tight.**

## Threat-model recommendations

1. **Harden the three open sudoers grants (HI1-HI3) before any wider deployment.** The unconditional NOPASSWD grants on `/usr/bin/env`, `/usr/bin/python3`, and `/usr/bin/cp` are the single highest-value target for a future bypass. Each can be tightened to the exact invocation used without changing any daemon code.

2. **Move `GrubSetKargs` out of a bash pipeline.** A dedicated setuid-root helper script or a systemd `ExecStart` unit would eliminate the three open sudoers grants above in one architectural move and would also make the operation testable without `bash -c` string construction.

3. **Implement the audit watermark sink (ME3) before production use.** Tail truncation is the least-detectable attack on the HMAC chain. A single `journald` NOTICE per transaction gives the SIEM an independent tip-of-chain reference at negligible cost.

4. **Add a global "mutating action in progress" gate (ME4) to prevent `UbuntuReleaseUpgrade` interleaving.** The 45-minute upgrade window is an unusually wide attack surface for dpkg-level interference; a simple in-memory mutex in `DaemonState` is sufficient.
